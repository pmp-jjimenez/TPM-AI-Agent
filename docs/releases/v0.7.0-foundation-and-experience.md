# TPM Operating System v0.7.0 — Foundation & Experience

## 1. Release overview

TPM Operating System v0.7.0 is the first demonstrable product release. Named
**Foundation & Experience**, it brings the repository's canonical program domain,
read-only service boundary, enterprise web application shell, Command Center,
Program Mission Control, design system, and product identity into one coherent
local product baseline.

This release is a foundation, not a production launch. It demonstrates how a
Technical Program Manager can move from a portfolio-level view into the recorded
operational state of a program. The web experience is read-only, AI behavior in
Mission Control is a preview, and the system still uses local single-user storage.

## 2. Product vision

TPM Operating System is an enterprise program intelligence product for Technical
Program Managers. Its purpose is to turn fragmented program facts into a clear,
traceable operating picture: what is happening, what needs attention, which
decisions matter, and where accountable action is required.

v0.7.0 establishes the product and engineering vocabulary needed to pursue that
vision without overstating the current intelligence layer. Stored program facts
remain distinct from generated guidance, and the frontend presents rather than
invents program state.

## 3. User value delivered

- A coherent enterprise application shell provides persistent navigation,
  context, product identity, and responsive behavior.
- The Home Command Center summarizes active local programs and their recorded
  risks, issues, dependencies, decisions, actions, and reported health.
- The Programs experience supports loading, empty, malformed-data, not-found,
  network-error, and retry states around the read-only API.
- Program Mission Control puts executive health, current priorities, canonical
  operational records, and AI-preview context into a single review surface.
- Consistent visual language, status semantics, focus treatment, and responsive
  layouts make the demonstrable product easier to scan and navigate.

## 4. Domain capabilities

The versioned, framework-neutral program domain supports these canonical records:

- **Programs** provide the aggregate boundary and program identity.
- **Actions** record accountable work with status, priority, ownership, dates,
  lifecycle, completion, and audit metadata.
- **Risks** record probability, impact, treatment priority, ownership, treatment,
  review, and acceptance data.
- **Issues** record active problems, severity, impact, ownership, dates, root
  cause, resolution, and closure metadata.
- **Dependencies** record internal or external commitments, ownership, status,
  type, required-by dates, impact, and mitigation.
- **Decision Records** preserve decision status, rationale, alternatives, dates,
  ownership, and impact.
- **Persona Routing** deterministically selects a primary and supporting expert
  context from explicit program and workflow signals. It is routing logic, not
  autonomous multi-agent execution.

Canonical entities use stable UUID identity, controlled vocabularies, validation,
legacy normalization, and explicit typed relationships. No relationships or
program conclusions are inferred.

## 5. Frontend experiences

### Application Shell

The React and Material UI application provides a desktop sidebar, mobile drawer,
top bar, route context, content container, and accessible navigation structure.
Executive Review and Operational Readiness appear as disabled navigation concepts;
they are not implemented web workflows.

### Home Command Center

The `/programs` route is the product home. It presents a time-aware welcome,
portfolio summary metrics calculated from loaded records, bounded focus and advisor
previews, and direct links to up to three active program cards. It does not provide
portfolio intelligence or cross-program AI analysis.

### Programs experience

The browser reads local program records through the FastAPI API, filters unusable
list entries defensively, sorts a copied list by name, and navigates using encoded
program identifiers. The API and web interface do not expose create, update, or
delete operations.

### Program Mission Control

The `/programs/:programId` route presents program identity, reported health,
confidence, lifecycle phase, record counts, deterministic priority signals, and
read-only sections for Risks, Issues, Dependencies, Decision Records, and Actions.
Priority signals are derived from stored records by transparent frontend rules;
they are not live AI analysis.

### Product Design System

A centralized theme defines the enterprise palette, typography, spacing, shape,
elevation, surfaces, borders, text roles, focus states, and semantic status and
confidence treatments. Shared badges and presentation primitives provide consistent
labels and safe unknown states.

### Responsive behavior and accessibility foundation

Layouts reflow across mobile, tablet, and desktop breakpoints; the permanent desktop
navigation becomes a temporary mobile drawer; content wraps to avoid horizontal
overflow. Semantic headings, landmarks, labels, keyboard-visible focus states,
text-plus-color status communication, and feedback roles establish an accessibility
foundation. This is not a claim of formal accessibility certification.

### Product identity

The shell presents **TPM Operating System** with the descriptor **Enterprise Program
Intelligence**, creating a consistent product identity across the demonstrable web
experience.

## 6. Architecture foundation

- `app/` contains the current Python domain, local persistence, CLI workflows,
  persona routing, reporting, SOW intake, and bounded intelligence services.
- `backend/api/` provides a read-only FastAPI transport with health, program-list,
  program-detail, and program-intelligence endpoints.
- `frontend/` contains the React, TypeScript, Vite, React Router, and Material UI
  browser application.
- `data/programs/` remains the local JSON persistence boundary.
- Workspace intelligence has a versioned structured response contract, bounded
  evidence references, strict validation, and deterministic fallback behavior.

The browser does not own persistence or backend reasoning. The API reuses the
existing application boundary rather than duplicating domain behavior. Authentication,
authorization, tenant isolation, a database, and production infrastructure are not
part of this release.

## 8. Known limitations

- AI Assessment is currently preview/placeholder behavior.
- Mission Control does not perform live AI analysis.
- The web interface has no editing workflows.
- Executive Review has not yet been redesigned.
- Major Incident War Room has not yet been implemented.
- Operational Readiness Center has not yet been implemented.
- Portfolio intelligence has not yet been implemented.
- The runtime is a single-user local development baseline.
- There is no production deployment yet.
- The read API has no authentication, authorization, tenant isolation, mutations,
  filtering, sorting, or pagination.
- Local JSON persistence has no database transactions or cross-process locking.

## 9. Deferred capabilities

Deferred work includes web editing and approval workflows, persistent intelligence
runs and feedback, executive review redesign, a Major Incident War Room, an
Operational Readiness Center, portfolio-level intelligence, richer program
artifacts, multi-user security, durable database persistence, production
observability, and deployment.

These are roadmap items, not hidden or partially delivered v0.7.0 features.

## 10. Roadmap toward v0.8 Intelligence

v0.8 **Intelligence** is the next product version. It will build on the trusted
domain and experience foundation to make program intelligence more useful,
grounded, explainable, and operationally reviewable. Likely product increments
include live Mission Control assessment, evidence-aware recommendations, explicit
decision support, and intelligence history, introduced only with clear boundaries
between stored fact, deterministic derivation, and model-generated guidance.

The next engineering sprint is **DX-1.0 Developer Console**. Its purpose is to
improve the local developer operating experience and provide a dependable console
for inspecting and working with the product foundation before intelligence scope
expands.

## 11. Demonstration flow

1. Start the FastAPI backend from the repository root.
2. Start the Vite frontend with `VITE_API_BASE_URL` set to the local API.
3. Open the application and identify the TPM Operating System / Enterprise Program
   Intelligence shell.
4. Review the Home Command Center welcome, portfolio metrics, focus preview, advisor
   preview, and active programs.
5. Select a program to enter Program Mission Control.
6. Review identity, phase, health, confidence, last update, and executive metrics.
7. Review Today's Priorities and explain that they are transparent signals derived
   from stored records.
8. Review the AI Assessment preview and explicitly note that live Mission Control
   analysis is deferred.
9. Walk through Risks, Issues, Dependencies, Decision Records, and Actions.
10. Resize to a narrow viewport to demonstrate responsive navigation and layout.

The demonstration should not imply editing, portfolio intelligence, production
deployment, or live Mission Control AI behavior.
